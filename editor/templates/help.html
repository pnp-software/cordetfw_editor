Cordet FW Editor - Online Help

Web-based editor to support the specification of a service-oriented software application

The editor provides an environment where the following items can be created, edited and linked with each other:

  • Specification Items
  • Verification Items
The editor explicityl supports the service concept of the Cordet Framework but can also be used independently of the framework as a generic requirements capture tool.

Editor Access

Access to the editor is for members only. Prospective users apply for membership by contacting the administrator. Members are given a user name and an initial password (which they are expected to change after the first log-in) and they are allocated to one or more projects.

Target Environment

The editor is aimed at deployment in a closed environment (e.g. company intranet) where a medium-sized team of trusted users cooperate to define the specification of a set of related applications.

Categories of Users

There are two types of users: the admin user and basic users.

The admin user is responsible for:

  • Creating and managing basic users
  • Creating projects and assigning their owner
  • Assigning basic users to a project
  • Defining the ValSets for the specification items of a project

The basic users (or just "users" for short) manage the project and the specification items they contain in accordance with their roles. Within a project, a user has one of two roles:

  • The Project Owner is appointed by the admin user when the project is created. The project owner can:
    • Create and edit applications within the project
    • Create new specification items and verification items within the project and its applications
    • Edit and delete the specification items and verification items he created
    • View and copy all specification items and verification items within the project and its applications
  • The Project User is allocated to a project by the admin user. The project user can:
    • Create new specification items and verification items within the project and its applications
    • Edit and delete the specification items and verification items he created
    • View and copy all specification items and verification items within the project and its applications

A user can be assigned (either as owner or as plain user) to multiple projects.

Projects

Projects are created by the admin user. A project is a container for one or more applications. The applications are created by the project owner. In order to support a history mechanism for specification and verification items, projects have a release number. The release number is initialized to zero when the project is created and is incremented by 1 when the project owner makes a new release of the project.

Applications

Applications belong to a project. They are created by the project owner. An application is a container for a subset of the specification items and verification items in a project.

In order to support a history mechanism for their specificaton and verification items applications have a release number. The release number has the form 'x.y'. Here, 'x' is the release number of the project to which the application belongs and 'y' is an non-negative integer which identifies the application release within a project release. The application release number is initialized to zero when the project is created or when a new release of the project is done and is incremented by 1 when the project owner makes a new release of the application.

Specification Items

Specification items describe a piece of information which specifies a feature of an application or of its enclosing project. A specification item could, for instance, consist of an application requirement, or of the definition of a data item, or of a model used to describe part of an application behaviour.

Within a project, a specification item is identified by the pair: domain:name. The domain is used to capture a set of logically related specification items. The name is unique within the domain.

Some of the attributes of a specification item have a fixed semantics. Others have a semantics which depends on the specification item category. The following categories of specification items are recognized by the editor:

  • Requirement: a requirement specifying an application
  • Data Item: a data item used in a project
  • Behavioural Model: a model of part of the behaviour of an application
  • Service: a service offered or used by applications in a project
  • Packet: a packet carrying a comment or a report
  • Packet Parameter: a parameter in a packet

The attributes of a specification item with fixed semantics are (asterisked attributes are mandatory; the others may be left empty):

  • Domain(*): the domain to which the specification item belongs
  • Name(*): the name of the specification item (unique within the domain)
  • Project(*): the project to which the specification item belongs
  • Application: the application to which the specification item belongs (empty if the specification item describes a project-level feature)
  • Category(*): the category of the specification item
  • Title(*): a short description of the specification item
  • Description: a long description of the specification item
  • Owner(*): the name of the user who created the specification item
  • Status(*): one of the following:
    • New: the item has been created in the current release
    • Modified: the item was created in a previous release and has been modified in the current release
    • Confirmed: the item was created in a previous release and has not been modified in the current release
    • Obsolete: the item was created in a previous release and has since been modified
    • Deleted: the item was created in a previous release and has since been deleted
  • Update Time(*): the time when the specification item was created or when the status of the specification was last updated
  • Previous Pointer: the pointer to the previous version of a specification item which has been modified (empty if the specification item has status "New")
  • Justification: the rationale for the existence of the specification item or for its value
  • Remarks: any remarks applicable to the specification item
  • ValSet: the ValSet

In the case of data items, the value can be defined either as a raw numerical value or as an arithmetic expression which uses the values of one or more data items.

Specification Item Types

Some categories of specification items have a type attribute. The range of type values is defined by the project owner. A type has the following attributes:

  • Category(*): the category to which the type applies
  • Project(*): the project for which the type is defined
  • Name(*): the name of the type
  • Description(*): description of the type
  • Native: the native type to which the type is mapped
  • Size: the size of the type in number of bits
  • Enumerated(*): a flag indicating whether the type is enumerated
  • Owner(*): the name of the user who created the specification item
  • Status(*): same as status of a specification item
  • Update Time(*): the time when the specification item was created or when the status of the specification item type was last updated
  • Previous Pointer: the pointer to the previous version of a specification item type which has been modified (empty if the specification item has status "New")
In the case of enumerated types, the type is defined by defining the (name, values) pair of the enumeration.

The "native" and "size" attributes support code generation for data item types in that they allow the types defined within the editor to be mapped to native types within an implementation language.

ValSets

The editor allows multiple set of values to be assigned to the same specification item. A set of values for the specification values is called a ValSet. The admin user defines the range of ValSets for each project. The 'Default' ValSet is pre-defined for all projects.

The ValSet attribute of a specification item is used to manage the ValSets: when the specification item is created, it is allocated to the Default ValSet. The user can then create new instances of the same specification item belonging to one of the other ValSets defined for the project to which the specification_item belongs.

The ValSet is managed as follows:

  • when a new specification item is created, its ValSet is set to 'Default'
  • When a specification item is edited, its ValSet remains unchanged
  • When a specification item is copied, the ValSet of the newly created copy can be set freely

Specification Item Operations

Specification items can be added, edited, copied and deleted.

When a specification item is added:

  • A new instance of the specification item is created
  • The Status of the specification item is set to 'New'
  • The Updated Time of the specification item is set to the current time
  • The Previous Pointer of the specification item is set to zero
  • All other attributes of the newly-created data item are set by the user through a form

The edit operation is only possible on specification items with status 'New', 'Modified' or 'Confirmed'. When a specification item of 'Confirmed' status is edited:

  • The status of the item instance is set to 'Obsolete' and a new instance of the specification item is created
  • The status of the new instance is set to 'Modified', its 'Previous Pointer' is set to point to the old instance and its other attributes are initialized to the same value as those of the old instance
  • The Updated Time of the new instance is set to the current time
  • Other attributes of the new instance can be updated by the user through a form

When, instead, a data item of 'New' or 'Modified' status is edited, then:

  • Its Status and Previous Pointer are kept unchanged
  • Its other attributes can be modified by the user through a form
  • The Updated Time of the new instance is set to the current time

The copy operation is only possible on specification items with status 'New', 'Modified' or 'Confirmed'. When a specification item is copied:

  • A new specification item instance is created with status 'New'
  • The Previous Pointer of the specification item is set to zero
  • The Updated Time of the specification item is set to the current time
  • The other attributes are initialized to the same value as those of the instance being copied and can be modified by the user through a form

The delete operation is only possible on specification items with status 'New', 'Modified' or 'Confirmed'. When a specification item with status 'Confirmed' or 'Modified' is deleted, its status is changed to 'Deleted'. When a specification item with status 'New' is deleted, the data item instance is deleted from the database.

When the project or application to which the specification item belongs is released, the status of 'New' and 'Modified' specification items is changed to 'Confirmed'.

Behavioural Models

Behavioural models are implemented as state machine or procedure models generated with the FW Profile Editor. The FW Profile Editor stores models in a database (the FW Profile Editor Database). Each model has an owner who is identified by its e-mail address. Within the FW Profile Editor, models have a name and can have a domain and a title. Models are defined as json strings and an svg representation which can be used to render them as pictures.

The Cordet FW Editor has a connection to FW Profile Editor Database and it allows a user to import all the models corresponding to one or more domains. In response to an import request, the Cordet FW Editor retrieves the FW Profile Models of the selected domains from the FW Profile Editor Database and imports them into a dedicated table (the FwProfileModel table) in the Cordet FW Editor Database. The import function processes each imported model according to the following rules:

  • If an entry exists in the FwProfileModel table with the same domain and name as the imported model, and if the status of this entry is 'CNF', and if the imported entry is identical to the local entry, then nothing is done.
  • If an entry exists in the FwProfileModel table with the same domain and name as the imported model, and if the status of this entry is 'CNF', and if the imported entry differs from the local entry, then the status of the local entry is changed from 'CNF' to 'OBS' and a new record is created in the FwProfileModel table to hold the imported model. The status of the newly created entry is set to 'NEW' and its 'previous' field is set to point to the old entry.
  • If an entry exists in the FwProfileModel table with the same domain and name as the imported model, and if the status of this entry is 'NEW', then the entry is overwritten with the imported data.
  • In all other cases, a new record is created in the FwProfileModel table to hold the imported model. The status of the newly created entry is set to 'NEW' and its 'previous' field is set to 'None'.
In the second and third case, an error is issued if more than one entry is found with the same domain:name:status combination is found.

Verification Items

Verification items are intended to define a piece of information which is used to verify a specification item. A verification item could, for instance, consist of a test case which verifies one or more requirements, or of a document which contains information which verifies the implementation of one or more specification items, or of a report of the review of the implementation of one or more specification items.

Some of the attributes of a verification item have a fixed semantics. Others have a semantics which depends on the verification item category. The following categories of specification items are recognized by the editor:

  • Test Case: a test case which verifies the implementation of some aspects of an application
  • Review: a review of the implementation of one or more specification items
  • Analysis: an analysis of the implementation of one or more specification items

The attributes of a specification item with fixed semantics are (asterisked attributes are mandatory; the others may be left empty):

  • Domain(*): the domain to which the verification item belongs
  • Name(*): the name of the verification item (unique within the domain)
  • Project(*): the project to which the verification item belongs
  • Application: the application to which the verification item belongs (empty if the specification item verifies a project-level feature)
  • Category(*): the category of the veriication item
  • Title(*): a short description of the verification item
  • Description: a long description of the verification item
  • Owner(*): the name of the user who created the specification item
  • Status(*): same as status of a specification item
  • Update Time(*): the time when the verification item was created or when the status of the verification item was last updated
  • Previous Pointer: the pointer to the previous version of a verification item which has been modified (empty if the specification item has status "New")
  • Remarks: any remarks applicable to the specification item

The attributes with a semantics which depends on the category of a verification item are listed in the table below. Each row in the table represents an attribute and each column defines the meaning of the attributes for a particular category of verification items.
Test Case Review Analysis
Pre-Conditions Pre-conditions for executing the test case Pre-conditions for the validity of the review Pre-conditions for the analysis of the review<
Expected Outcome Expected outcome of test case Expected outcome of review Expected outcome of analysis
Actual Outcome Actual outcome of test case Actual outcome of review Actual outcome of analysis
To each verification item, one or more verification links may be attached. A verification link identifies a specification item which is verified by the verification item. A verification link includes a description of the conditions under which the target of the link is verified by the verification item.
